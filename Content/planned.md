## üõ†Ô∏è Module 0 ‚Äî Local & Remote State Setup

### Decisions

* **Create S3 bucket for Terraform state**
  *Why:* Think of this like a shared notebook where Terraform writes down what it has built. Storing it in S3 means you can access it from anywhere and share it across your team.

* **Enable versioning on the bucket**
  *Why:* If Terraform‚Äôs ‚Äúnotebook‚Äù gets messed up, versioning is your backup ‚Äî you can roll back to a previous version.

* **Create DynamoDB table for state locking**
  *Why:* This is like putting a ‚ÄúDo Not Disturb‚Äù sign on the notebook. It stops two people from editing the same infrastructure at the same time.

* **Name resources using project + environment**
  *Why:* Makes them easy to identify in the AWS Console (e.g., `tfstate-employee-directory-dev`).

### Variables

* `state_bucket_name` ‚Äî Name of S3 bucket for storing Terraform state.
* `state_dynamodb_table` ‚Äî Name of DynamoDB table for locking.
* `aws_region` ‚Äî Region where bucket and table will be created.
* `tags` ‚Äî Default tags to apply to both resources.

### Docs to Read (Why)

* **aws\_s3\_bucket** ‚Äì Learn syntax for creating S3 buckets in Terraform.
* **aws\_s3\_bucket\_versioning** ‚Äì Enable versioning for rollback safety.
* **aws\_dynamodb\_table** ‚Äì Create the table Terraform uses for locking.
* **terraform backend s3** ‚Äì Configure Terraform to use the bucket/table automatically.

### AI Prompt Template

> Generate Terraform configuration that creates:
>
> 1. An S3 bucket `${var.state_bucket_name}` in `${var.aws_region}` with versioning enabled, tagged with `${var.tags}`.
> 2. A DynamoDB table `${var.state_dynamodb_table}` with primary key `LockID` (string) for state locking, tagged with `${var.tags}`.
> 3. Configure Terraform backend to use this S3 bucket for remote state storage and DynamoDB for locking. Use variables for names, region, and tags.

## üåç Module 1 ‚Äî Global Project Setup

### Decisions

  *Why:* S3 is the **filing cabinet** for Terraform‚Äôs memory. DynamoDB is the **‚ÄúDo Not Disturb‚Äù sign** so only one change happens at a time.

### Docs to Read (Why)

* **terraform state locking dynamodb** ‚Äì Stops accidental simultaneous edits.
* **terraform workspaces** ‚Äì Switch between dev/prod without rewriting code.
* **terraform variables / outputs / locals** ‚Äì Keep code clean and reusable.

  *Why:* Like a receptionist that sends visitors to the right desk.

  *Why:* The ALB‚Äôs checklist to know if your app is healthy.
